Feasible & Low Effort

Drag & Drop Files Into the Editor

Analysis of adding file drag-and-drop to Just Artifacts — your single-page code preview tool.

The short answer: this is easy.

You're looking at roughly 30–50 lines of JavaScript and a handful of CSS rules. The HTML5 Drag & Drop API combined with the File API is a mature, well-supported browser standard. Your code editor is a plain <textarea>, so populating it with file contents is a single assignment. No framework changes, no build step, no dependencies.

The complexity isn't in reading the file — it's in deciding what UX you want around it. That's where the real decisions live.

How It Works

1

Listen for dragover

Prevent the browser's default behavior (which is to open the file in a new tab). This activates the drop zone.

2

Listen for drop

Extract DataTransfer.files from the event. This gives you a FileList of dropped items.

3

Read with FileReader

Call reader.readAsText(file) on each file. The onload callback gives you the raw string content.

4

Populate the editor

Set codeEditor.value = content, trigger updatePreview(), optionally auto-detect language from file extension.

Core Code

Here's the minimal implementation — the actual logic is trivial:

// Attach to the entire page so dropping anywhere works
document.addEventListener('dragover', (e) => {
    e.preventDefault();
    // Show visual feedback here (border flash, overlay, etc.)
});

document.addEventListener('drop', (e) => {
    e.preventDefault();
    const files = e.dataTransfer.files;
    if (files.length === 0) return;

    const file = files[0];
    const reader = new FileReader();

    reader.onload = () => {
        elements.codeEditor.value = reader.result;
        updatePreview();
    };

    reader.readAsText(file);
});

That's it. ~20 lines. The rest is UX polish.

UX Decisions

🎯

Drop Zone Scope

Drop anywhere on the page, or only on the code editor? Anywhere is more intuitive and forgiving — users don't need to aim precisely.

👁️

Visual Feedback

When a file hovers over the window, show a border highlight or subtle overlay. Without it, users won't know the drop was registered until after it happens.

🏷️

Auto-Detect Language

.html → HTML, .jsx / .tsx → React. Nice-to-have that makes the feature feel smart without being complex.

⚠️

Unsaved Changes

If the editor has unsaved content, should dropping a file overwrite silently, or prompt first? Recommend: overwrite silently — it's a file drop, the user is being intentional.

📁

Multiple Files

What if someone drops 3 files? Options: take the first, concatenate them, or reject. Recommend: take the first, simplest behavior.

📐

File Size Limit

Don't try to read a 50MB log file into a textarea. A soft cap at ~1MB with a toast notification keeps things sane.

Visual Feedback Concept

When a file is dragged over the window, the code editor panel could highlight like this:

Drop file here

.html, .jsx, .tsx, .css, .js, .txt

A dashed border + green tint on the code panel. Clean, obvious, disappears on drop or drag-leave.

Edge Cases & Handling

Scenario Handling Effort
Binary file (image, PDF) readAsText returns garbage. Check file.type or extension before reading. easy
Very large file (>1MB) Check file.size before reading. Show a brief warning toast. easy
Multiple files dropped Take files[0]. Ignore the rest. Simple and predictable. easy
File with no extension Default to HTML mode. User can switch manually if needed. easy
Encoding issues (non-UTF8) readAsText defaults to UTF-8. Rare edge case for this use case. skip
Dragging from browser (not file system) Check e.dataTransfer.types — if it includes "Files", it's a file drop. easy

Implementation Plan

Add dragover + drop listeners on document

Prevent default, extract file, read as text, populate editor.

Add visual feedback on dragover / dragleave

Toggle a CSS class on the code editor panel — dashed border + subtle background tint.

Add file type validation

Only accept text-based files. Reject binaries with a brief inline message.

Auto-detect language from extension

Map .html → HTML, .jsx/.tsx → React. Update the language select.

Verify: drop an HTML file, check preview updates

Test with .html, .jsx, .txt, and a binary file (image) to confirm rejection works.

Language Auto-Detection

A small mapping that makes the feature feel polished:

const LANG_MAP = {
    '.html': 'html',
    '.htm':  'html',
    '.jsx':  'react',
    '.tsx':  'react',
    '.css':  'html',   // CSS works in HTML mode
    '.js':   'html',   // JS wrapped in <script> works too
    '.txt':  'html',
};

function detectLanguage(filename) {
    const ext = '.' + filename.split('.').pop().toLowerCase();
    return LANG_MAP[ext] || 'html';
}

Analysis Through Four Lenses

⚙️

Technical

HTML5 Drag & Drop + File API. Universal browser support. ~30-50 lines of JS. Zero dependencies. No build changes needed.

🧠

Cognitive

Drag-and-drop is a universal OS metaphor. Users will understand it instantly. Visual feedback during drag is the only thing that needs design attention.

🏗️

Structural

Purely additive. No existing code needs refactoring. Event listeners on document. CSS class toggle for visual state. Blast radius: near zero.

♿

Accessible

D&D isn't keyboard-accessible, but that's fine — paste still works. This is an enhancement, not a replacement. No existing accessibility is degraded.

Verdict: Go for it. Low risk, high utility, ~1 hour of work including polish.
